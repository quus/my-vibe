# mv-setup — Project Bootstrap & Credentials

> 모든 mv-* 스킬은 **Jira/GitHub 연결정보**를 전제로 동작한다.
> 새 프로젝트에서 *처음 1회* 실행해 환경을 세팅하고, 그 뒤로는 *변경된 값만* 갱신.

## 1. Triggers
- "초기 설정"
- "setup my-vibe"
- "credentials 설정"
- "jira 연결"
- "github 연결"
- "환경변수 설정"
- "mv setup"

## 2. Phase
`setup` — 모든 다른 mv-* 스킬의 *선행 요건*. (`mv-feature-upsert` 이전)

## 3. Inputs
대화형(AskUserQuestion) 또는 사전에 정의된 env-vars 사용. 받는 항목:

### 3.1 Jira (필수 — 4개)
| 키 | 예시 | 비고 |
|---|---|---|
| `JIRA_BASE_URL` | `https://acme.atlassian.net` | 끝에 `/` 없이. https만 |
| `JIRA_EMAIL` | `user@example.com` | Jira 계정 이메일 |
| `JIRA_API_TOKEN` | `ATATT...` | id.atlassian.com → Security → API tokens |
| `JIRA_PROJECT_KEY` | `SI` | 프로젝트 키(대문자 2~10자) |

### 3.2 GitHub (인증 방식 1개 선택)
| 방식 | 키 | 비고 |
|---|---|---|
| SSH (권장) | (없음) | `~/.ssh/id_*` 키가 GitHub에 등록되어 있어야. 테스트만 |
| PAT (HTTPS) | `GITHUB_TOKEN` | `repo`, `workflow` 스코프. `gh auth login`로도 가능 |
| `gh` CLI | (없음) | `gh auth status`로 검증 |

추가:
| 키 | 예시 | 비고 |
|---|---|---|
| `GITHUB_REPO` | `quus/my-product` | owner/repo 형식 |
| `GITHUB_DEFAULT_BRANCH` | `main` | 기본 `main` |

## 4. Procedure

### Step A — 상태 점검 + 기존 값 로드
1. 현재 CWD가 git 레포인지 확인. 아니면 *주의 알림*("이 디렉터리는 git이 아닙니다 — 그래도 진행할까요?").
2. 기존 `.env`, `.env.example`, `.gitignore` 존재 여부 점검.
3. **기존 `.env`가 있으면 모든 값을 *메모리로 로드*** 해 다음 단계의 *기본값(default)*으로 사용한다.
   - 로드 순서 우선순위: 1) `.env` 파일, 2) 현재 셸 환경변수, 3) 비어 있음.
   - 시크릿 값은 *마스킹 처리* 후 표시(`****ABCD` 형식, 마지막 4자만).
4. **로드된 값을 단일 표로 한 번에 보여준 뒤** 사용자에게 안내:
   *"각 항목에서 엔터만 누르면 기존 값을 그대로 유지합니다. 새 값을 입력하면 덮어씁니다."*

### Step B — Jira 정보 수집 (대화형, 기존값-유지 모드)
각 항목을 `AskUserQuestion` 또는 `read -p "Key [기존값-마스킹]: "` 형식으로:

```
JIRA_BASE_URL     [https://acme.atlassian.net]:           # 엔터 → 기존 유지
JIRA_EMAIL        [user@example.com]:                     # 엔터 → 기존 유지
JIRA_API_TOKEN    [****ABCD (유지) / 새로 입력]:          # 엔터 → 기존 유지, 입력 시 검증
JIRA_PROJECT_KEY  [SI]:                                   # 엔터 → 기존 유지
```

**기존값-유지 로직** (모든 필드 공통):
1. 입력이 *빈 문자열*(엔터만) → 메모리의 기존 값 그대로 사용.
2. 입력이 *있으면* → 형식 검증 → 통과 시 새 값으로 교체, 실패 시 *재입력 루프*(기존 값 보존).
3. 기존 값이 *없고* 입력도 *빈 문자열* → 필수 항목이면 다시 요청, 선택 항목이면 빈 채로 유지.
4. 입력 단계에서는 *연결 검증을 하지 않음* — 모든 값 확정 후 Step C에서 일괄 검증.

검증 규칙(입력값이 있을 때만):
- `JIRA_BASE_URL`: `^https://[^/]+$` (https 필수, 끝 슬래시 금지).
- `JIRA_EMAIL`: 이메일 형식.
- `JIRA_API_TOKEN`: **stdin masking**. 결과 로그에도 마스킹.
- `JIRA_PROJECT_KEY`: `^[A-Z][A-Z0-9_]{1,9}$`.

### Step C — Jira 연결 검증
```bash
curl -sS -u "$JIRA_EMAIL:$JIRA_API_TOKEN" \
  "$JIRA_BASE_URL/rest/api/3/myself" | jq -r '.accountId, .displayName'
```
- 200 OK + accountId 반환 → 통과.
- 401/403 → 토큰 또는 이메일 오타.
- 404 → BASE_URL 잘못.

그리고 PROJECT_KEY 검증:
```bash
curl -sS -u "$JIRA_EMAIL:$JIRA_API_TOKEN" \
  "$JIRA_BASE_URL/rest/api/3/project/$JIRA_PROJECT_KEY" | jq -r '.key, .name'
```
- 200 + 일치하는 key → 통과.

실패 시 사람에게 정확한 에러 + 가능한 원인 표시 후 *해당 항목 재입력* 루프.

### Step D — GitHub 인증 선택 & 검증 (기존값-유지 모드)
인증 방식 선택 (`AskUserQuestion`) — 기존 `GITHUB_AUTH_METHOD`가 있으면 *해당 옵션을 첫 번째이자 기본*으로 노출:
- **SSH** (기본 권장): `ssh -T git@github.com 2>&1` → "Hi <user>!" 나오면 통과.
- **PAT** (HTTPS): `GITHUB_TOKEN` 검증
  ```bash
  curl -sS -H "Authorization: token $GITHUB_TOKEN" \
    https://api.github.com/user | jq -r .login
  ```
- **gh CLI**: `gh auth status` 출력 확인 (gh 미설치면 PAT/SSH로 안내).

각 필드 입력도 *기존값-유지* 동일하게:
```
GITHUB_AUTH_METHOD       [ssh]:                      # 엔터 → 기존 유지
GITHUB_TOKEN             [****WXYZ (유지) / 새로]:   # PAT 모드일 때만, 엔터 → 기존 유지
GITHUB_REPO              [quus/my-product]:          # 엔터 → 기존 유지
GITHUB_DEFAULT_BRANCH    [main]:                     # 엔터 → 기존 유지
```

`GITHUB_REPO` 검증(입력값 있을 때만):
- 형식: `^[a-zA-Z0-9_.-]+/[a-zA-Z0-9_.-]+$`.
- 접근 확인:
  - SSH: `git ls-remote git@github.com:$GITHUB_REPO.git HEAD`
  - PAT: `curl -sS -H "Authorization: token $GITHUB_TOKEN" https://api.github.com/repos/$GITHUB_REPO | jq -r .full_name`

### Step E — 파일 생성 (멱등)

#### `.gitignore` 갱신
`.env`가 *이미* `.gitignore`에 있는지 grep. 없으면 추가:
```
# Local environment (never commit)
.env
.env.local
.env.*.local
```

#### `.env` (gitignored, 실제 값)
```ini
# my-vibe — local credentials (NEVER COMMIT)
# generated by mv-setup at YYYY-MM-DDTHH:MM:SSZ
JIRA_BASE_URL=https://acme.atlassian.net
JIRA_EMAIL=user@example.com
JIRA_API_TOKEN=ATATT...
JIRA_PROJECT_KEY=SI

GITHUB_REPO=quus/my-product
GITHUB_DEFAULT_BRANCH=main
# 다음 줄 중 1개만 활성 — SSH 사용 시 GITHUB_TOKEN 비워둠.
# GITHUB_TOKEN=ghp_...
GITHUB_AUTH_METHOD=ssh   # ssh | pat | gh
```

권한: `chmod 600 .env`.

#### `.env.example` (committed, placeholder만)
```ini
# my-vibe — credentials template. Fill values into .env (which is gitignored).
JIRA_BASE_URL=https://<workspace>.atlassian.net
JIRA_EMAIL=
JIRA_API_TOKEN=
JIRA_PROJECT_KEY=

GITHUB_REPO=<owner>/<repo>
GITHUB_DEFAULT_BRANCH=main
# GITHUB_TOKEN=
GITHUB_AUTH_METHOD=ssh
```

### Step F — Shell 통합 안내
사용자에게 *지금 세션*에서 활성화 명령 출력:
```bash
set -a; source .env; set +a
```
혹은 영구 적용을 위해:
- **direnv** 권장: `echo 'dotenv' > .envrc && direnv allow`
- **수동**: `~/.zshrc`/`~/.bashrc`에 `export $(grep -v '^#' /path/to/.env | xargs)`

## 5. Guardrails

- **토큰 절대 git 커밋 금지** — `.env`가 `.gitignore`에 있는지 *쓰기 직전* 재확인.
- **토큰 마스킹** — 모든 출력에서 마지막 4자만 노출(`****ABCD`).
- **HTTPS만** — JIRA_BASE_URL은 http:// 거부.
- **재실행 안전** — 기존 `.env`가 있으면 *모든 값을 기본값으로 로드*. 백업(`.env.bak.<timestamp>`) 후 새 `.env` 쓰기.
- **엔터-유지 원칙** — 모든 입력에서 *빈 입력*은 기존 값 유지로 해석. *새 값을 명시한 필드만* 변경. 결과 보고서에 *변경된 필드 목록*과 *유지된 필드 목록*을 구분 표시.
- **실패 시 부분 쓰기 금지** — 모든 검증이 통과해야 `.env` 쓰기.
- **시크릿을 채팅 출력에 표시 금지** — 모델이 응답 본문에 토큰을 다시 인쇄하지 않도록 주의.

## 6. Cost & Time
- 토큰: ~5k (sonnet, 단순 I/O + curl)
- 시간: 2~5분 (대화형 입력 시간 포함)

## 7. When NOT to use
- 자격 증명이 이미 `direnv` / 1Password CLI / Vault로 관리되는 환경 — 그 경로 그대로 사용 권장.
- 비밀 관리 정책상 `.env` 파일 금지 환경 — 대신 *vault read*로 환경에 주입 후 검증만 수행.

## 8. Chains
- 선행: (없음 — 가장 앞)
- 후행: `mv-feature-upsert`로 자연스럽게 연결. 보고서 §Next steps에 명시.

## 9. 비밀 관리 모범 사례

| 등급 | 보관 위치 |
|---|---|
| 🔒 Production | Vault, AWS Secrets Manager, GCP Secret Manager |
| 🔐 Team-shared | 1Password Business / Doppler |
| 👤 Personal dev | `.env` (gitignored) + direnv |
| ❌ 금지 | git commit, Slack, 평문 wiki, `~/.bashrc`(공유 기기) |

본 스킬은 *Personal dev* 등급을 기본으로 한다. 팀 표준이 다르면 해당 도구의 *주입 후 검증* 단계로 대체.
